
1. 什么是索引
    索引是一种排好序的数据结构，用于提升数据检索的效率。
    如果数据行和索引是存储在一起的，就是聚簇索引，表中只能有一个聚簇索引，如果数据行和索引是分开存储的，则是非聚簇索引。
    以我们常用的Mysql来说，表一般会有一个主键。 表的主键索引，就是聚簇索引，而其他索引则是非聚簇索引。
    因为Mysql索引底层使用的是B+树，B+树的特点是非叶子节点只存储索引，不存储数据，而叶子节点既存储索引，还存储数据。
    而主键的叶子节点存储了索引值+完整的数据行，而其他索引存储的是索引值+主键值（非完整数据行）。
2. 索引底层数据结构
    二叉树 层高比较高，而且针对自增列，会退化成链表
    红黑树 层高比较高
    Hash表 hash碰撞 不支持区间查询
    B树  非叶子节点也有数据
    B+树 B树的一种。非叶子节点只存储索引，不存储数据。叶子节点之间用指针相连，提高区间访问的性能。
3. 什么是回表
    回表（Table Lookup）是Mysql中一种查询优化现象，主要发生在使用二级索引（非聚簇索引）进行查询时。
    当查询条件基于二级索引，但返回的字段不在二级索引中时，即非覆盖索引，Mysql需要通过主键值回溯到主键索引（聚簇索引）获取完整数据行，这个过程称为回表
    针对非主键索引（二级索引）：
        覆盖索引：索引包含所有查询字段，无需回表
        非覆盖索引：查询的字段不在索引中，需要回表，增加I/O消耗

6. 聚集索引与覆盖索引与索引下推到底是什么
    聚集索引：数据行与索引存储在一起，表中仅允许一个聚集索引
    覆盖索引：查询所需字段完全包含在索引中，无需回表
    非覆盖索引：查询的字段不在索引中，需要回表，增加I/O消耗
    索引下推：将WHERE条件下推到存储引擎层执行，减少回表次数，优化非覆盖索引查询，提升性能，仅在MySQL 5.6+版本支持。
            SELECT * FROM users WHERE name = '张三' AND age > 25;  -- 使用ICP（Index Condition Pushdown索引下推）优化 索引为name和age的联合索引
            查询时，索引扫描过程中直接过滤 age > 25，减少回表次数。
            索引下推：根据索引找到记录-》在存储引擎层直接应用索引列的WHERE条件-》对满足条件的记录回表
            传统索引查询：根据索引找到记录-》回表查询完整记录-》在服务层应用WHERE条件过滤
7. 联合索引底层数据存储结构是怎样的
    联合索引也是索引，所以也是排好序的数据结构。和单字段索引的区别在于，联合索引的排序是有类似于层级或者是优先级的概念，比如 有个用户表user，字段有id,name,age，
    该表有个联合索引name_age_idx，该索引的排序逻辑则为先按照name排序，name相同，则按照age排序。
8. 如何使用mysql执行计划定位线上慢sql
    1. ‌关注 type 字段
        system: 系统表，数据量极少
        const: 通过主键或唯一索引等值查询，只扫描一行
        eq_ref: 唯一索引等值扫描，常用于连接查询
        ref: 非唯一索引等值扫描
        range: 范围扫描，如 BETWEEN，>, <
        index: 索引扫描
        ALL: 全表扫描
    2. ‌关注 key 和 possible_keys
        key：实际使用的索引。如果为null，则表示未使用索引
        possible_keys: 可能使用的索引
    3. ‌关注 rows 和 Extra
        rows: 优化器预估需要扫描的行数，数值越小越好。
        Extra: 列包含额外信息，如 Using index 表示使用了覆盖索引，Using where 表示 WHERE 条件未被索引覆盖。
9. mysql最左前缀优化原则实现原理详解
    Mysql联合索引（复合索引）使用的关键策略，通过从索引的最左列开始匹配查询条件，提升查询效率。
    索引失效场景：
        查询条件未包含最左列
        最左列使用函数或表达式 如 date_format(最左列)=?，lower(最左列) = ?等
10. 为什么推荐使用自增整型的主键而不是UUID
    存储空间占用：自增整型4-8字节（int,bigint）,比UUID的36字节节省空间
    查询效率：自增整型的查询效率较快。
            因为字符串比较大小是一个一个字符比较的，以16为UUID为列，如果前面15位相同，则需要比较16次。
            而且自增整型，可以利用B+树的顺序存储特性，范围查询仅需少量的磁盘IO
    写入效率：自增整型直接追加到索引尾部，避免页分裂，提升写入效率。而UUID插入需要重新排序，增加页分裂风险，降低写入效率。

    分库分表；
        分库分表时，主键不能使用自增主键，因为如果使用自增主键时，不同的库中主键可能重复，所以一般使用的是雪花算法生成主键ID。
        雪花算法虽然不是严格自增的，但是是趋势递增的。

12.Mysql的NOT IN除了会导致索引失效，还会导致什么？
    NOT IN会导致索引失效
    NOT IN不能查询到为NULL的数据
        假如有个user表，有3条记录，name分别为Tom,Jerry,NULL，则
        select * from user where name not in ('Tom');
        返回结果为Jerry行，不会返回NULL行
    NOT IN中不能使用NULL
        select * frm user where name not in ('Tom', NULL);
        返回结果为空，因为在MYSQL中，NULL属于未知值，只能使用IS NULL或者IS NOT NULL判断，不能使用=，<>，IN, NOT IN等判断

18.事务有哪些特性？靠什么保证
    事务的特性ACID
    原子性 Atomicity 一个事务中的操作要么全部成功，要么全部失败
        通过undo log保证，记录来需要回滚的日志信息，事务回滚时，通过相反的sql撤销已经执行成功的sql
    一致性 数据库总是从一个一致性的状态转换到另一个一致性的状态
        由其他三大特性保证
    隔离性 Isolation 一个事务的修改结果在最终提交前，对其他事务是不可见的
        由MVCC保证
    持久性 Durable 事务一旦提交，则所做的修改就永久保存到数据库中了
        通过redo log + bin log保证
19.如何保证Mysql和Redis缓存的双写一致性
    不可用：先删除缓存，后更新数据库
        在删除缓存后，更新数据库前，其他线程来请求数据，发现缓存中不存在，则从数据库中获取，并放入缓存中，此时缓存中为旧数据，
        然后本线程更新数据库后，此时数据库和缓存中数据出现永久不一致。
    先更新数据库，再删除缓存
        更新数据库后，准备删除缓存前，其他线程获取的到的数据还是从缓存中获取的旧数据，数据会有短暂的不一致，
        但在删除缓存后，有新的请求进来时，会发现缓存不存在，从数据库中获取到最新数据，并放入缓存，此时缓存和数据库中一致
    延时双删 先删除缓存，再更新数据库，再删除缓存
    异步消息队列：
        写操作 -> MySQL更新 -> 消息队列发送 -> Redis异步更新
20.多级缓存不一致问题
    Mysql + redis + 本地缓存
    一般是保证最终一致性。先更新数据库，在删除缓存，然后使用消息队列发送消息，清除本地缓存
20.Mysql的锁
    基于锁的属性分类：共享锁，排它锁
    基于锁的粒度分类：表锁（Innodb,Myisam），行锁(Innodb)，记录锁，间隙锁，临键锁
    基于锁的状态分类：意向共享锁，意向排它锁

    共享锁：又称读锁，当一个事务为数据加上读锁后，其他事务只能对该数据加读锁，不能加写锁。
        主要是为了支持并发的读取数据，读取数据的时候不支持修改，避免出现不可重复读的问题
    排它锁：又称写锁。当一个事务为数据加上写锁后，其他事务不能对该数据加任何锁。
        主要是为了在数据修改的时候，不允许其他事务同时修改，也不允许其他事务读取，避免出现脏数据和脏读的问题

23.Mysql创建索引会遵循哪些原则
    其实就是使用mysql索引时，有哪些注意点。
    1.最好是全职匹配
    2.遵循最左匹配原则，避免索引失效
    3.索引列上别计算
    4.针对非主键索引，最好不要使用select *，避免回表
    5.Like语句 %写在最右边，例如 name like 'tom%'，这种可以走索引
    6.Varchar加引号，避免隐式转换
        正例：SELECT * FROM USER WHERE sex = '1';
        反例：SELECT * FROM USER WHERE sex = 1; -- 隐式转换为数字，而sex字段是varchar，导致索引失效

25.意向锁是啥(ix意向共享锁，ix意向排它锁)
    意向锁是mysql InnoDB存储引擎中的一种特殊表级锁，用于协调行锁和表锁之间的关系。
    它的作用主要是避免在加锁时，需要逐行检查行锁是否存在，从而提升数据库的并发性能。

    核心特点：
    1.表级锁：意向锁是表级别的锁，但它只表示事务有意向在表中的某些行上加锁，而不是真正锁定这些行。
    2.自动管理：意向锁由InnoDB引擎自动添加和管理，用户无法手动操作
    3，不与行锁冲突：意向锁与行级的共享锁（S锁）和排它锁（X锁）是兼容的，不会阻塞行锁的申请

    意向锁的类型：
        意向锁分为两种类型：
            意向共享锁（is）: 表示事务打算在表中的某些行上加上共享锁（读锁）
            意向排它锁（ix）：表示事务打算在表中的某些行上加上排它锁（写锁）

    意向锁的工作机制：
        当一个事务需要对表中的某一行或多行加锁时，InnoDB会先为该表加上相应的意向锁（IS或IX），然后再对具体的行加锁。

    意向锁的使用场景（作用）：
    1.优化锁冲突检测：当事务想要对整个表加锁时，只需要检查表上是否存在意向锁，而无需遍历所有行来判断是否已有行锁
    2.支持多粒度锁共存：意向锁使得Innodb能够同时支持表锁和行锁，避免了在高并发下因锁冲突导致的性能问题
    3.避免死锁：通过提前判断表级锁的状态，可以有效减少冲突和死锁的发生

    意向锁的兼容性：
    1.意向锁之间是互相兼容的，例如IS锁和IX锁可以共存
    2.意向锁与表级锁（共享锁S和排它锁X）是互斥的，这以为着表上如果存在IS或IX锁，就不能再获取表级的S锁和X锁。

    总结：意向锁就是Mysql InnoDB存储引擎中的一种特殊表级锁，用于协调行锁和表锁之间的关系。
        它只表示事务有意向在表中的某些行上加锁，而不是真正锁定这些行。
        它由InnoDB引擎自动添加和管理的，用户无法手动操作。
        它是在事务给表加行锁时，自动给表加意向锁。
        这样的好处是，当事务需要给表加表级锁时，只需要检查表上是否存在意向锁，不需要逐行检查行锁是否存在，从而提升数据库的并发性能。
        通过提前判断表级锁的状态，可以有效减少冲突和死锁的发生。
