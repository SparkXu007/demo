
1.什么是redis?
    redis是一个基于内存的键值数据库，是一种非关系型数据库，性能高而且数据结构丰富。
    因为redis的数据存储在内存中，所以读写速度极快。
    redis支持String,List,Set,Hash,ZSet5种数据类型。
    String:最基本的数据类型，可以存储文本，数字或者二进制数据，最大支持512MB。
    List:有序的字符串集合，支持从两端插入和弹出元素，从用于实现消息队列
    Set:无序的字符串集合，成员唯一，适合用于标签系统或者抽奖活动
    Hash:键值对集合，适合存储对象，如用户信息
    ZSet:有序集合，带有分数的集合，成员按分数排序，常用于排行榜
2.Redis持久化机制
    RDB:
        工作原理：定期生成全量数据的二进制快照（.rdb文件）
        触发方式：
            1.手动执行save(阻塞)或bgsave(后台异步)
            2.配置自动触发（如 save 900 1表示900秒内之少一个键被修改）
            3.执行shutdown或flushAll时自动触发
        核心流程（bgsave）：
            1.通过fork()创建子进程
            2.通过子进程将数据写入临时rdb文件，不阻塞主进程
            3.替换旧的rdb文件
        优势：文件小，恢复快
        劣势：快照间隔内数据可能丢失
    AOF:
        工作原理：记录每一条写操作命令（.aof文件）
        写入策略：
            always:每个写命令都同步刷盘（最安全，性能最低）
            everySec:每秒同步一次（默认推荐，平衡安全与性能，有可能丢失1秒的数据）
            no:由操作系统决定何时同步（性能最高，可能丢失数据）
        重写机制：
            1.解决aof文件膨胀问题
            2.通过bgRewriteAof创建新aof文件，移除冗余命令
        优势：支持everySec/always刷盘，数据一致性高
        劣势：文件大，恢复速度慢

        重写触发机制：
            1.自动触发：通过配置 auto-aof-rewrite-percentage(增长比例 默认100%)
                    和 auto-aof-rewrite-min-size(最小文件大小 默认64MB)，超过阈值则触发重写
            2.手动触发：执行bgRewriteAof命令
            3.启动时触发：如果aof文件损坏，redis启动时会尝试重写
        核心思想：
            通过读取当前数据状态，生成一个新的、更紧凑的aof文件，替代原有庞大的aof文件。
            新文件只包含重建当前数据集的最少命令，消除冗余命令（如 多次set同一个key）
        详细流程：
            1.主进程fork一个子进程执行重写任务，避免阻塞主线程
    RDB+AOF：兼顾RDB的快速恢复和AOF的少丢数据，生产首选
3.Redis集群
    主从集群：一个主节点写数据，一个或多个从节点读数据，主节点会把数据同步到从节点。
        优点：保证读写分离，提升性能，又能够备份数据
        缺点：如果主节点宕机了，需要手动切换主节点，无法自动故障转移
    哨兵集群：基于主从集群，增加哨兵节点，实现 主从监控+自动故障转移
        优点：主节点宕机后，不需要手动切换主节点，而是哨兵节点通过选举，将一个从节点切换成主节点，解决了单点故障问题
        缺点：所有节点都存储了全量数据，无数据分片能力，存在容量瓶颈
    Cluster集群：通过16384个槽位分片，将这些槽位分给集群中不同的主节点，每个主节点只负责保管一部分数据，
            然后给每个主节点配置从节点，备份数据
        优点：解决了单点故障问题，实现了自动故障转移，解决了容量瓶颈问题
        Redis Cluster集群通过内置的节点间通信机制实现自动故障转移，无需依赖外部哨兵监控
4.Redis删除策略（过期数据清理）
    3种删除策略。
    定时删除：
        触发机制：为键设置独立计时器（如 expire key 60），到期时立即执行删除操作
        优点：内存释放及时（秒级清理），内存利用率高
        缺点：CPU消耗大（每秒触发多次），可能阻塞redis服务器
        适用场景：实时性高的场景（如 实时交易系统），需确保数据及时更新
    定期删除
        触发机制：Redis定期（每秒10次，默认 hz=10）随机扫描过期键
        优点：通过控制扫描的频率和时长，平衡CPU和内存占用
        缺点：无法立即释放过期键（可能保留10秒数据），内存占用稍高
        适用场景：内存紧张场景（如 缓存系统）
    惰性删除
        触发机制：当客户端访问键时，检查键是否过期，过期则删除
        优点：CPU友好（仅在访问时占用CPU），避免定时任务压力
        缺点：可能长时间保留过期键（若 未访问），内存浪费
        适用场景：CPU约束场景（如 高并发读写），减少CPU开销

    一般是定期删除和惰性删除组合使用，定期删除 清理冷数据，惰性删除 清理热数据（高频访问的数据）
    # 配置定期删除频率
    hz 10  # 每秒扫描 10 次
    # 配置惰性删除（默认开启）
    lazyfree-lazy-eviction yes
4.Redis数据淘汰策略（内存不足时清理）
    当内存达到maxmemory时，触发淘汰策略
    8种淘汰策略。
    无淘汰：
        noeviction 默认策略 拒绝写入新数据，只允许读取和删除操作。
        新写入会报错 OOM command not allowed when use memory
        适用场景：对数据一致性要求较高，不允许丢失任何数据的场景（需确保内存足够或有其他扩容机制）
    基于访问频率（LRU/LFU）：
        allkeys-lru: 从所有的key中，淘汰最近最少使用的key（最久未被使用的优先淘汰,只看使用时间的先后顺序，跟数量关系不大）
        volatile-lru: 仅从设置了过期时间的key中，淘汰最近最少使用的key
        例如：假设redis的内存只够写入5个key，如果按顺序写入了A，B,C,D,E5个key，
            如果此时往redis中写入第6个键F,则会将A淘汰，此时redis中则存储的是B,C,D,E,F
            如果接着往redis中写入键G，则会将B淘汰，此时redis中则存储的是C,D,E,F,G，
            如果接着往redis中写入C,则只会更新C,
            如果接着往redis中写入键H，则会将D淘汰，而不会淘汰C，因为C被再次使用了，使用的时间相当于排到来后面，所以此时redis中则存储的是C,E,F,G,H

        allkeys-lfu: 从所有的key中，淘汰最不经常使用的key（访问频率最低的优先淘汰，只看数量，和时间先后关系不大）
        volatile-lfu: 从设置了过期时间的key中，淘汰最不经常使用的key
        例如：假设redis的内存只够写入5个key，且此时redis中存储的数据为A，B,C,D,E5个key，在最近一段时间内，其中A,B,C,D,E的访问次数分别为50，10，30，40，20，
            如果此时往redis中写入第6个键F,则会将B淘汰，因为B的访问次数最小
    随机淘汰：
        allkeys-random: 从所有的key中，随机淘汰key
        volatile-random: 从设置了过期时间的key中，随机淘汰key
    过期时间优先：
        volatile-ttl: 从设置了过期时间的key中，淘汰 剩余存活时间最短的key
5.Redis淘汰算法
    LRU（最近最少使用）,LFU（最不经常使用）,随机淘汰
6.缓存击穿，缓存穿透，缓存雪崩
    缓存击穿：缓存中不存在，数据库中存在。
        原因：热点数据不存在于缓存中 或者 热点key失效，导致大量访问直接打到数据库。
        解决方案：
            1.热点数据预热；
            2.互斥锁：当缓存失效时，只允许一个线程去数据库加载数据并更新缓存，其余请求等待
            3.逻辑过期：缓存永不过期，但在缓存中存储一个逻辑过期时间，当一个请求访问缓存时，如果发现数据的逻辑过期时间已到，
                    系统会启动一个后台线程去数据库中查找最新的数据，并更新缓存。而当前请求会继续使用旧数据，知道新数据完成加载
            4.设置热点数据永不过期：由后台异步更新缓存内容，适用于数据不常变更的场景。
    缓存穿透：缓存和数据库中都不存在。
        原因：
            1.恶意攻击
            2.业务设计缺陷：未对请求参数做合法性校验
        解决方案：
            1.可以缓存空值：对于查询结果为空的数据也进行缓存，并设置过期时间（如5分钟）
            2.参数校验：在接口层对请求参数进行合法性校验，过滤非法请求
            3.布隆过滤器：在访问缓存前，先通过布隆过滤器判断数据是否存在，避免无效请求进入数据库
    缓存雪崩：
        原因：
            1.大量缓存数据在同一时间集体失效，
            2.冷启动未预热，
            3.Redis服务整体宕机
            导致大量请求直接访问数据库，造成数据库压力过大甚至崩溃。
        解决方案：
            1.设置随机过期时间
            2.缓存预热：‌在系统启动或大促前，提前加载热点数据；
            3.高可用缓存集群：‌ 使用 Redis 主从、哨兵或集群模式，避免单点故障；
            4.熔断限流机制：‌ 当数据库压力过大时，自动限流或熔断，保护后端服务。

9.Redis大key问题
    什么是redis大key？
        String类型：value体积 > 10MB
        集合类型（Hash,List,Set,ZSet）：成员数量 > 5000

        大key的危害：
            性能瓶颈：单线程模型下，操作大key耗时过长，阻塞主线程。
            持久化困境：RDB/AOF持久化耗时增加，可能引发数据不一致
            网络传输隐患：大Key传输易导致带宽饱和，影响分布式系统稳定性
    怎么定位？
        # 扫描全库，输出最大Key信息
        redis-cli --bigkeys
        # 查看指定Key的内存使用情况
        MEMORY USAGE your_key
        # 查看Key的序列化长度
        OBJECT ENCODING your_key
    怎么解决？
        拆分法：将大key拆分为多个小key
    怎么预防？
        1.定期执行 -- bigkeys检测
        2.在设计阶段就避免，将大key拆分为多个小key。
