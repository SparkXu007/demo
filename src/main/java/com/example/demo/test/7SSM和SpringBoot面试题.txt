
1.Spring是什么
    Spring是一个开源的轻量级容器框架。核心就是我们常说的面向切面（AOP）和控制翻转（IOC）.
    IOC通过容器管理对象生命周期，简化对象的创建与管理
    AOP通过切面横切逻辑。
    IOC的实现机制有两种，
        一种是依赖注入，即通过构造器注入、Setter注入或字段注入，容器自动装配依赖。
        // Setter注入
        @Autowired
        public void setB(B b) {
            this.b = b;
        }
        还有一种是Bean工厂管理，即容器维护Bean池，按需创建和销毁对象
        // Bean定义
        @Bean
        public A a(B b) {
            return new A(b);
        }
    AOP的实现机制是通过动态代理。即通过JDK动态代理或cglib代理实现切面。
    // JDK动态代理
    Proxy.newProxyInstance(loader, interfaces, handler);
    ‌通知类型‌：前置通知（@Before）、后置通知（@After）、环绕通知（@Around）等。

    AOP在项目中用的最常见是日志记录。比如申明一个日志注解，然后在需要记录日志的方法上（如新增、修改、删除）使用该注解，
    然后在创建一个切面类LogAspect，在类里面实现日志的逻辑。

4.Spring事务的实现方式和原理以及隔离级别
    实现方式有两种，一种是编程式的，一种是申明式的。
    编程式：
        手动控制事务的开启、提交和回滚，即手动调用transactionManager的getTransaction()开启事务，commit()提交事务,rollback()回滚事务
        // 编程方式
        TransactionStatus status = transactionManager.getTransaction(new DefaultTransactionDefinition());
        try {
            // 业务逻辑
            transactionManager.commit(status);
        } catch (Exception e) {
            transactionManager.rollback(status);
        }
    申明式：
        通过注解@Transaction申明或xml配置，自动管理事务
        // 注解方式
        @Transactional
        public void transferMoney() {
            // 业务逻辑
        }

    底层原理即AOP代理机制和事务管理器定义事务操作。
    Spring通过AOP创建事务代理对象，拦截方法调用。
    通过PlatformTransactionManager接口定义事务操作（如getTransaction、commit、rollback）。

    事务隔离级别：有5种
    DEFAULT: 使用数据库默认的隔离级别（如Mysql的REPEATABLE_READ可重复读）
    READ_UNCOMMITTED: 读未提交（脏读）
    READ_COMMITTED: 读已提交（防止脏读，但不可重复读）
    REPEATABLE_READ: 可重复读（可能产生幻读）
    SERIALIZABLE: 串行化。最高隔离级别，防止所有并发问题，但性能低

    Spring事务隔离级别优先于数据库配置，当事务隔离级别不一致时，实际执行级别以Spring配置为准。
5.Spring的事务传播机制
    Spring事务主要通过@Transaction注解的propagation属性控制，共有7种。
    REQUIRED: 默认行为，若存在事务则加入，否则创建新事物（最常用）。这种是最常用的，适用于大多数业务场景。
    SUPPORTS: 若存在事务则加入，否则以非事务方式执行（事务可选）。适用于事务可选的辅助操作
    MANDATORY: 必须在事务中执行，否则抛出异常（强制事务）。适用于事务必须存在的场景
    REQUIRES_NEW: 创建新事务，挂起当前事务（独立执行）。适用于日志记录、异步操作
    NOT_SUPPORTED: 以非事务的方式执行，挂起当前事务（事务无关操作）。适用于无需事务的场景
    NEVER: 以非事务方式执行，若存在事务，则抛出异常（事务冲突）。适用于严格禁止事务的场景。
    NESTED: 嵌套事务，若存在事务，则在子事务中执行，否则创建新事务（独立回滚）。复杂业务的细粒度控制
            父事务回滚，子事务也会回滚；子事务回滚，父事务不一定回滚，如在父事务中捕获了子事务抛出的异常，则父事务不会回滚
6.Spring事务失效的场景
    1.异常类型不匹配
        默认仅回滚RunTimeException和Error，检查异常（如IOException）不会触发回滚。
        解决方案：
            使用rollback显示指定异常类型即可
            @Transactional(rollbackFor = Exception.class)
            public void myMethod() throws Exception {
                // 业务逻辑
            }
            或者转换为运行时异常。
    2.异常被捕获但未抛出
        解决方案：
            继续抛出异常
            try {
                // 业务逻辑
            } catch (Exception e) {
                throw new RuntimeException(e);
            }
            或者手动设置回滚状态
            TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
    3.传播行为配置错误
        解决方案：
            检查传播行为，如设置 propagation = Propagation.REQUIRED
            @Transactional(propagation = Propagation.REQUIRED) // 默认值

    4.内部方法调用
        同一个类中，方法A内调用方法B，Spring AOP代理失效
        解决方案：
            通过注入自身实现代理调用
            @Autowired
            private MyService self;
            self.myTransactionalMethod();
    5.方式是非public的
        解决方案：
            修改方法访问权限为public
    6.数据库不支持事务
        使用了不支持事务的数据库引擎（如MyISAM）
        解决方案：
            切换支持事务的引擎（如InnoDB）

9.Springboot的自动配置原理
    通过约定优于配置的原则，简化了Spring应用的配置和部署过程。
    主要依赖三个核心，即@SpringBootApplication复合注解，spring.factories文件以及一系列的条件注解@ConditionalOnxx。
    @SpringBootApplication复合注解
        包含了@SpringBootConfiguration,@EnableAutoConfiguration,@ComponentScan三个注解，
        其中@SpringBootConfiguration注解表明该类是Springboot的配置类，
        @ComponentScan注解则表明启动组件扫描，自动发现并注册当前包及其子包下带有@Component，@Service,@Repository等注解的bean
        @EnableAutoConfiguration注解则是开启自动装配的关键。
    当我们启动一个SpringBoot项目时，@EnableAutoConfiguration注解便被激活，该注解本身也是一个复合注解，其核心是通过@Import注解
    导入了AutoConfigurationImportSelector类。这个类是一个选择器，负责决定应该加载哪些配置类。
    它会调用SpringFactoriesLoader.loadFactoryNames方法，在类路径下寻找所有META-INF/spring.factories文件，
    然后从这些spring.factories文件中，读取EnableAutoConfiguration这个key对应的所有自动配置类的全限定类名列表。
    这些配置类一般位于spring-boot-autoConfiguration这个核心jar包中。
    获取到的这些配置类列表并不是全部加载，而是通过选择器类AutoConfigurationImportSelector基于一系列的条件注解
    （@ConditionalOnClass,@ConditionalOnBean,@ConditionalOnProperty等）对这些配置类进行筛选，只有满足所有条件的配置类，
    才会最终被实例化并应用到当前的Spring上下文中。
    总结为一下7步，
    1.启动应用：执行main方法，启动SpringApplication
    2.激活注解：扫描到@SpringBoorApplication注解，进而激活@EnableAutoConfiguration注解
    3.调用选择器：@EnableAutoConfiguration注解会通过@Import注解导入AutoConfigurationImportSelector类
    4.加载候选列表：AutoConfigurationImportSelector类会从所有的META-INF/spring.factories中读取EnableAutoConfiguration
                这个key对应的所有自动配置类的全限定类型列表
    5.条件筛选：基于一系列的条件注解（）对这些配置类进行筛选
    6.注册bean：将最终筛选出的自动配置类加载到Spring容器中，执行其中的@Bean方法，创建并注册相应的Bean实例
    7.完成装配：应用程序上下文初始化完成，所有必要的组件已就绪，应用可以对外提供服务。
