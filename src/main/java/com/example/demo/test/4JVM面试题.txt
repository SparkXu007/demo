
1.JVM内存模型
    JVM内存包括堆、栈、方法区（元空间）、本地方法栈、程序计数器。
        堆：对象
        栈：局部变量
            如果局部变量是对象，类似于user，则引用执行对象，也就是堆。
        方法区：常量、静态变量、类信息
        本地方法栈：本地方法（native,底层是C++实现的）
        程序计数器：下一步要执行的代码的位置
                    当该线程执行的过程中，如果存在优先级更高的线程时，操作系统将切换到优先级更高的线程执行，
                    而当执行完毕又切换回该线程时，程序要继续执行，所以程序计数器存储的是下一步要执行的代码的位置。
    其中堆、方法区（元空间）是线程共享的，而栈、本地方法栈、程序计数器是线程私有的。
    每个栈都有一个栈帧，栈帧包括 局部变量表、操作数栈、动态链接、方法出口
            比如申明一个方法内部的局部变量int i = 1;
    堆包括 年轻代（1/3）和老年代(2/3)，比例为1:2。而年轻代又分为Eden区和两个S区，比例为8:1:1。
            对象晋升：
                1.分代年龄阈值触发。
                    对象在年轻代经历15次Minor GC后仍未被回收，则分代年龄达到阈值，则对象从年轻代进入老年代。
                    分代年龄阈值可配置：可通过-XX:MaxTenuringThreshold调整
                2.大对象直接进入老年代
                    对象大小超过-XX:PretenureSizeThreshold阈值（默认0，表示未启用）。
                3.Survivor区空间不足
                    Minor GC后，存活对象无法全部放入Survivor区，超出部分直接进入老年代
                4.动态年龄判断
                    Survivor区中，相同分代年龄对象总大小超过50%时，所有分代年龄大于等于该分代年龄的对象直接进入老年代
            minor GC: 针对年轻代做垃圾收集，频繁但停顿时间短（毫秒级）。
                触发条件：
                    1.Eden区满了则触发minor GC。
                可达性分析算法：GC Root。线程的本地变量、静态变量
            full GC: 针对整个堆和元空间做垃圾收集，停顿时间长（秒级），显著影响系统响应。
                触发条件：
                    1.老年代满了则触发full GC
                    2.元空间满了也触发full GC
                    3.显示调用system.gc()，有可能触发full GC
            优化建议：避免full GC，减少STW的影响。
            具体措施：比如合理增加老年代和元空间的大小（如-Xmx、-XX:MaxMetaspaceSize）。
            ‌监控指标‌：
            jstat -gc <pid>：实时监控GC频率和停顿时间。
            jmap -heap <pid>：查看堆内存使用情况。

2.JVM垃圾收集机制
    STW机制
3.STW机制是啥
    STW即Stop-The-World，是JVM垃圾回收过程中的一种机制，指的是在垃圾回收的过程中，JVM会暂停所有应用线程（即停止所有正在执行的java代码）。
    其目的是确保垃圾回收器能够安全、准确的执行内存标记和清理操作，而不会受到应用线程的干扰。
4.垃圾收集器有哪些？设计原理及区别
    Serial收集器：
        特点：单线程执行，采用复制算法进行新生代回收，标记整理算法进行老年代回收
        适用场景：适用于客户端模式下的小型应用，内存较小的环境
        优势：简单高效，没有线程切换开销
    Parallel集器（Parallel Scavenge收集器，Parallel Old收集器）
        特点：多线程并行回收，注重吞吐量，采用复制算法
        适用场景：适用于后台计算密集型任务，如大数据处理，批量数据导出等
        优势：追求高吞吐量，适合对响应时间要求不高的场景
    CMS收集器（Concurrent Mark Sweep）
        特点：并发执行，追求低延迟，采用标记清除算法
        适用场景：适用于对响应时间要求较高的应用，如Web服务，在线交易系统等
        优势：减少停顿时间，但存在内存碎片问题
    G1收集器（Garbage First）
        特点：将堆划分为多个大小相等的region，采用标记整理和复制算法
        适用场景：适用于大堆内存（通常超过6GB）环境，追求可预测的停顿时间
        优势：支持可预测的停顿时间，避免Full GC，适合大规模应用
    ZGC收集器
        特点：亚毫秒级停顿时间，支持TB级堆内存，采用标记复制算法
        适用场景：适用于需要极低延迟的大型应用，如实时交易系统，大型在线游戏等。
        优势‌：停顿时间极短，适合超大规模内存应用。
    ‌小内存应用‌：选择Serial或ParNew收集器。
    ‌高吞吐量需求‌：选择Parallel Scavenge或Parallel Old收集器。
    ‌低延迟需求‌：选择CMS或G1收集器。
    ‌超大堆内存‌：选择G1或ZGC收集器。
    ‌实时系统‌：选择ZGC或Shenandoah收集器。
5.垃圾回收算法
    标记清除算法
        标记阶段：从GC Root触发，遍历对象引用链，标记所有可达的对象
        清除阶段：回收未标记对象占用的内存

        特点：实现简单，但易产生内存碎片
            需两次扫描，标记存活对象和清除垃圾对象
            适用于老年代（对象存活率高）
    标记复制算法
        将可用内存分为大小相等的两块，每次只使用其中一块，当这一块内存用完了，就将还活着的对象复制到另外一块上面，再把已使用的那块清除掉
        优点：实现简单，不会产生内存碎片
        缺点：将内存缩小为原来的一半，代价高昂
        适用于新生代的S区
    标记整理算法
        标记阶段和标记清除算法，在标记后，将存活的对象压缩到内存的一端，然后清理边界外空间
        优点：不会产生内存碎片，充分利用内存
        缺点：与标记清除算法相比，多了对象的移动，所以性能相对较慢
        适用于老年代
    分代收集算法：
        将堆分为新生代和老年代，根据对象的生命周期选择算法
        新生代使用复制算法，老年代使用标记清除或者标记整理算法
18.GC如何判断对象是否可以回收
    引用计数法：
        每个对象有一个引用计数器，新增一个引用时，计数器+1，引用失效时，计数器-1，计数器为0时，可以回收
        优点：效率高，只用判断引用计数器的值是否为0即可
        缺点：无法回收循环引用的对象。例如A依赖B，B依赖A，导致两者都不能回收，导致内存泄漏。
    可达性分析算法（三色标记算法）：
        ‌白色‌：未被标记对象
        ‌灰色‌：已标记但未完全扫描引用对象
        ‌黑色‌：已标记且引用扫描完成。
        从GC Root开始向下搜索，搜索所走过的路径被称为引用链。当一个对象到GC Root没有任何引用链相连，则证明此对象不可用，则可以回收。

        GC Root:
            虚拟机栈（栈帧中的本地变量表）中引用的变量
            方法区中的类静态变量引用的对象
            方法区中常量引用的对象
            本地方法栈中的JNI(即Native方法)引用的对象

        可达性分析算法中的不可达对象并不是立即死亡的，对象拥有一次自我拯救的机会。
        对象被系统宣告死亡至少要经过两次标记过程：
            第一次是经过可达性分析算法后，分析发现对象没有与GC Root相连接的引用，则检查对象是否重写了Finalize()方法，
                若重写了finalize()方法，则放入F-queue队列，由Finalizer线程执行finalize()方法。
                若未重写，直接标记为可回收
            第二次是在finalize()方法执行后，若对象未重新建立引用，则标记为可回收，否则对象复活，不可回收。

            当复活后的对象在再次不可达时，会重复上面的流程，可能标记为回收，也可能再次复活。

        浮动垃圾（多标）：
            原因：在并发标记阶段，垃圾收集器（GC线程）正在扫描对象图，而用户线程（业务线程）也在运行，可能会修改对象之间的引用关系。
                1.假设对象A引用对象B,GC线程已将A标记为黑色（即其所有引用都扫描完成）
                2.此时用户线程执行A.b = null，断开了A对B的引用
                3.此时，B应该成为垃圾，但由于B已经被标记为灰色（GC线程已访问过），它不会被再次扫描，因此在本轮GC中不会被回收
                这就是浮动垃圾，也就是所谓的多标问题。本应回收的垃圾没有被回收。
            解决方案：
               等待下一轮GC重新标记回收即可。
        漏标问题：
            原因：在并发标记阶段，垃圾收集器（GC线程）正在扫描对象图，而用户线程（业务线程）也在运行，可能会修改对象之间的引用关系。
               1.假设GC已经完成了标记，且对象A,C为黑色，B为白色
               2.此时用户线程执行A.b = 1，增加了A对B的引用
               3.此时，B不应该成为垃圾，但由于A已经被标记为黑色（GC线程已访问过且其所有引用都扫描完成），它不会被再次扫描，
               因此在本轮GC中B会被错误的回收
               这就是漏标问题。本该保留的对象被回收了。
        解决方案
            CMS 回收器‌：使用‌增量更新‌（Incremental Update），当黑色对象新增对白色对象的引用时，将该黑色对象重新标记为灰色，在重新标记阶段再次扫描。
            G1 回收器‌：使用‌原始快照‌（SATB），记录并发阶段的引用关系快照，确保即使引用被删除，也能在最终标记阶段重新扫描。

6.JVM调优参数
    1. 堆内存控制
        -Xms：初始堆大小（如-Xms4g），建议与-Xmx相同，避免动态扩容导致性能抖动。
        -Xmx：最大堆大小（如-Xmx4g），需根据应用需求设置，避免OOM异常。
        -Xmn：新生代大小（如-Xmn20m），通常设为堆内存的1/3或1/4。
    2. 元空间控制
        -XX:MetaspaceSize：元空间初始大小（如256m）。
        -XX:MaxMetaspaceSize：元空间最大大小（如512m），防止类加载器泄漏导致OOM。
        . 垃圾回收器选择
        -XX:+UseG1GC：启用G1垃圾回收器，适合大堆内存应用。
        -XX:MaxGCPauseMillis：设置G1目标最大停顿时间（如200ms）。
    4. 新生代与老年代比例
        -XX:NewRatio：新生代与老年代比例（如-XX:NewRatio=2，新生代占1/3）。
        -XX:SurvivorRatio：Eden与Survivor比例（如-XX:SurvivorRatio=8，Eden占8/10）。
    5. 其他关键参数
        -XX:+PrintGCDetails：输出GC详细信息，用于分析GC日志。
        -XX:+HeapDumpOnOutOfMemoryError：OOM时生成堆转储快照。
    调优建议
       堆内存设置：生产环境建议-Xms和-Xmx相同，值设为可用物理内存的70-80%。
       元空间限制：必须设置MaxMetaspaceSize，防止内存泄漏。
       垃圾回收器选择：根据应用需求选择合适的GC（如G1适合大堆内存应用）。
       性能监控：使用JVM监控工具（如JVisualVM）分析GC日志和性能瓶颈。

       java -Xms4g -Xmx4g -Xmn20m -XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m -XX:+UseG1GC -XX:MaxGCPauseMillis=200 -XX:+PrintGCDetails -XX:+HeapDumpOnOutOfMemoryError -jar your-app.jar
7.如果JVM频繁minor GC，该如何解决？
    修改JVM配置，调整新生代的内存大小 -Xmn：新生代大小（如-Xmn20m），通常设为堆内存的1/3或1/4。
7.如果JVM频繁Full GC，该如何解决？
    Full GC触发条件：
        1.老年代满了则触发full GC
        2.元空间满了也触发full GC
        3.显示调用system.gc()，有可能触发full GC

    检查是否有大量对象长时间存活
    检查是否存在内存泄漏

    使用 jstat 查看GC情况：
        jstat -gcutil <pid> 1000
    使用 jmap 查看堆内存使用情况
        jmap -heap <pid>

8.JVM OOM如何排查？
    1.看报错日志，定位是哪种类型的OOM.
        java.lang.OutOfMemoryError: Java heap space：堆内存溢出。
        java.lang.OutOfMemoryError: GC overhead limit exceeded：GC开销过大。
        java.lang.OutOfMemoryError: Metaspace：元空间溢出。
        java.lang.OutOfMemoryError: Direct buffer memory：直接内存溢出。
    2.如果配置了堆转储，则分析堆转储文件
        -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/path/to/dump/
    3.堆内存溢出（Java heap space）
        原因‌：对象过多或内存泄漏
        排查方法‌：
        检查是否有大对象分配，如大数组、大集合。
        检查是否存在内存泄漏，如未释放的资源（如File、Connection等）。
        检查是否频繁创建对象，使用对象池或缓存机制。
        调整JVM堆内存大小（-Xmx 和 -Xms）。
      GC开销过大（GC overhead limit exceeded）
        原因‌：频繁GC但回收的内存很少
        排查方法‌：
        检查是否有大量对象长时间存活。
        检查是否存在内存泄漏。
        调整GC策略，如使用G1GC。
        增加堆内存大小。
      元空间溢出（Metaspace）
        原因‌：加载的类过多或类加载器泄露。
        ‌排查方法‌：
        检查是否有大量动态生成的类（如CGLib代理类）。
        检查类加载器是否正确释放。
        调整元空间大小（-XX:MaxMetaspaceSize）。
      直接内存溢出（Direct buffer memory）
        原因‌：直接内存使用超出限制。
        ‌排查方法‌：
        检查是否使用了NIO（如Netty、Jetty）。
        检查是否频繁创建Direct ByteBuffer。
        调整直接内存大小（-XX:MaxDirectMemorySize）。
    总结：先定位是哪种类型OOM，然后分析错误日志，定位到问题位置。如果是代码逻辑问题，则修改代码，如果是配置问题，则调整配置

9.CPU占用100%，标准排查步骤
    1.查看系统的负载。使用系统监控工具，比如top查看系统的的负载情况，快速定位高负载的进程
        # 1.找到cpu耗用最厉害的 进程PID
        top
        # 2.通过进程PID找到CPU耗用最厉害的 线程PID
        top -H -p 进程PID
        # 3.通过下面的命令，将第2步的线程PID转为16进制
        printf '0x%x\n' 线程PID
        # 4.使用jstack查看线程错误信息
        jstack 进程PID | grep 16进制线程PID -A 20
    2.确定是哪个进程导致CPU高占用
    3.查看日志文件。检查是否有异常或错误信息
    4.检查代码。看代码是否存在性能问题
    5.查看数据库的查询性能

9.内存逃逸

    未逃逸，方法逃逸，线程逃逸
    未逃逸的的对象分配在栈中，方法逃逸和线程逃逸的对象内存分配在堆中。

    未逃逸：
    public void test() {
        User user = new User();
        user.setName("spark");
    }
    User对象未逃逸，分配在栈中

    方法逃逸：
    public User test() {
        User user = new User();
        user.setName("spark");

        return user;
    }
    返回了User对象，逃逸了，对象内存分配在堆中。
