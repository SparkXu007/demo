
1.如何使用自定义注解实现防重复提交
    自定义注解+AOP的方式
    先定义自定义注解类，例如RepeatSubmit

    @Target(ElementType.METHOD)
    @Retention(RetentionPolicy.RUNTIME)
    @Documented
    public @interface RepeatSubmit {
        enum Type {PARAM, TOKEN};

        TYPE limitType() default TYPE.PARAM;

        long lockTime() default 5;
    }
    然后在需要防重复提交的方法上使用该注解
    然后再定义一个AOP类，用于处理防重复提交逻辑

3.40亿QQ号如何去重？内存1G.
    目前最长的qq号应该是10位数字，而int类型的数据范围最大为21亿多，也是10位数字，假设int类型满足存储qq条件,
    每个int类型是占4个字节，40亿QQ号则需要 4,000,000,000 * 4个字节存储，转换为G则为
    4,000,000,000 * 4 / 1024 / 1024 / 1024 =14.9G，超过了内存为1G的限制。
    而如果使用位图的话，每一位代表一个qq号是否存在（0不存在，1存在），而40亿qq号则最多需要40亿位，转换为G则为
    4,000,000,000 / 8 / 1024 / 1024 / 1024 = 0.5G，满足条件。
    然后遍历位图，将所有位置为1的索引值收集起来就是去重后的QQ号。
4.百万级别数据的excel如何快速导入到数据库中
    使用easyExcel分批读取数据，比如每次读取2000条记录，然后做入库处理。避免一次加载excel中所有数据和数据一次入库。
    如果excel有多个sheet页，可以考虑使用多线程。
    启用多线程，每个线程读取指定的一个或者多个sheet页数据，每个线程内，每读取一定的记录数，比如说2000条，做一次入库处理，
    避免大数据一次入库。
    当然，这里只考虑了最理想的场景，实际场景中可能存在一些校验等，还有可能数据校验通过，入库失败等一些场景，还有就是是否可以重复导入（重复导入是覆盖还是报错等）
    最好在表中加一个批次字段，比如batch_no，如果导入数据部分成功，可以根据批次号删除数据，然后重新导入。
    还有就是增加批次记录表以及批次记录详情表，批次表主要记录批次操作的结果，成功，失败，部分成功等，批次详情表则记录失败的原因等

7.当给第三方提供接口时，需要注意哪些事?
    需要提供接口文档。接口设计时，需要注意
    1.提供的接口遵循RESTful规范，数据格式统一使用json格式。
    2.考虑安全性。
        1.确保调用方身份：使用API key，OAuth 2.0，JWT等确保调用方身份。
        2.敏感数据加密或者脱敏：接口返回的敏感数据（手机号、身份证号等）需加密或脱敏
        3.参数校验：校验参数类型、格式等
        4.还可以根据实际情况，考虑是否需要增加限流：比如按照ip，用户，或者接口维度限制QPS
    3.考虑接口性能
        1.看能否使用缓存，合理设置缓存过期时间
        2.看能否使用异步处理。根据实际情况，看能否使用异步线程或者消息队列等异步处理
    4.日志记录
        记录接口的请求和响应数据，便于定位（可以存日志文件或者存表）
8.设计redis的key和value时，有哪些原则
    key:
        1.可读性与规范性
        2.避免key过长
        3.控制key的数量
    value：
        1.选择合理的数据结构
            比如缓存用户信息，可以使用Hash,统计排行榜/排序，使用zSet等
        2.避免大value
        3.设置合适的过期时间
